# Codex Guidelines

このドキュメントは、このリポジトリに対して Codex がコードやドキュメントを修正する際の行動ルールを定義するものです。

Codex の役割は **既存アーキテクチャと規約を尊重しながら、安全で最小限の変更を行うこと**です。

Codex は「正確で、小さく、テスト可能な変更」を最優先とします。

---

# Codexの役割

Codex の主な責務は以下です。

- ソースコードの実装
- バグ修正
- テストコードの作成
- 必要に応じたドキュメント更新

Codex は **プロジェクト管理やタスク優先順位の判断は行いません。**

タスク管理や進行管理は **Claude Code が担当します。**

---

# 基本原則

Codex は以下の原則を必ず守ること。

## 1. 変更は最小限にする

変更は可能な限り小さくする。

以下は禁止。

- 関係ないコードのリファクタリング
- 不必要なファイル構造変更
- 無関係なモジュールの変更

以下を優先する。

- 局所的な修正
- 小さな差分
- 対象箇所のみの変更

---

## 2. 既存アーキテクチャを尊重する

新しい設計パターンを導入しない。

既存のレイヤ構造に従う。

例

- controller
- service
- repository
- domain

レイヤを飛ばしてアクセスしてはいけない。

悪い例

controller → repository

正しい例

controller → service → repository

---

## 3. 新しい依存ライブラリを追加しない

新規ライブラリの導入は禁止。

既存のライブラリやユーティリティを優先して利用する。

例外は明示的に指示された場合のみ。

---

## 4. 公開インターフェースを変更しない

以下の変更は禁止。

- APIレスポンス形式
- 公開関数のシグネチャ
- データベーススキーマ

明確な指示がある場合のみ変更可能。

破壊的変更は行わない。

---

# コード変更ルール

コードを変更する際は以下を守る。

1. 必要最小限の変更を特定する  
2. 関係するファイルのみを変更する  
3. 既存の命名規則を維持する  
4. プロジェクトのコーディングルールに従う  

ファイル全体の書き換えは禁止。

---

# エラーハンドリング

新しいロジックには必ず適切なエラーハンドリングを追加する。

例

- 入力値のバリデーション
- null / undefined のチェック
- 適切な例外処理

エラーを無視する実装は禁止。

---

# ログ出力

重要な処理を追加する場合はログを追加する。

注意事項

- 既存のログフレームワークを使用する
- 機密情報はログに出力しない

ログ出力禁止例

- パスワード
- トークン
- 個人情報

---

# セキュリティ要件

コード変更時は必ずセキュリティを考慮する。

特に注意する箇所

- 認証処理
- 認可チェック
- データベースクエリ
- ファイルアップロード
- コマンド実行

防ぐべき脆弱性

- SQLインジェクション
- コマンドインジェクション
- 権限昇格

---

# テスト要件

コード変更には可能な限りテストを追加する。

テスト種別

- Unit Test
- Integration Test

テストには以下を含める。

- 正常系
- 境界値
- 異常系

単に実行できるだけのテストは禁止。

ロジック検証を目的とする。

---

# ドキュメント更新

コードの挙動が変更された場合はドキュメントを更新する。

対象例

- README
- API仕様
- 設定ドキュメント

コードとドキュメントの整合性を保つ。

---

# コードスタイル

プロジェクトのスタイルガイドに従う。

例

- 命名規則の統一
- 関数は小さく保つ
- 可読性の高いコードを書く
- 深いネストを避ける

フォーマッタやリンタが存在する場合はそれに従う。

---

# Pull Requestの準備

変更を提出する前に以下を確認する。

- テストが成功する
- ビルドが成功する
- lintチェックが通る
- フォーマットが適用されている

PRは小さく保つ。

推奨サイズ

200行以内

最大

500行以内

---

# 禁止事項

Codex は以下を行ってはいけない。

- 大規模なコード書き換え
- アーキテクチャ変更
- CI設定変更
- セキュリティ関連モジュールの変更（指示なし）
- 新規ライブラリ追加
- 既存テストの削除

---

# 判断に迷った場合

要件が不明確な場合は以下を優先する。

- 最小限の変更
- 既存挙動の維持
- 安全性の確保

創造性よりも **正確性と安全性** を優先する。
